# DATA_SPECIFICATION_v1.md

# Transcriptomic Data QC & Analysis Framework

## Purpose

This document defines all input and output data formats used in the project.

The goal is to ensure that the Data Cleaner, Analysis Engine, reporting module and Streamlit interface use consistent file structures.

---

## 1. Input Data

### 1.1 Expression Matrix

Purpose:

Stores gene expression values for all biological samples.

Supported formats:

- CSV
- TSV
- XLSX

Supported orientations:

- sample x gene
- gene x sample

Supported expression values:

- raw counts
- normalized counts
- TPM
- FPKM
- log-transformed expression values

Example (sample x gene):

sample_id,TP53,KRAS,EGFR
S1,5.2,7.1,2.3
S2,6.1,8.4,1.9
S3,5.8,7.9,2.1

Example (gene x sample):

gene,S1,S2,S3
TP53,5.2,6.1,5.8
KRAS,7.1,8.4,7.9
EGFR,2.3,1.9,2.1

Validation Rules:

- file must contain at least one sample
- file must contain at least one gene
- expression values should be numeric
- duplicated gene identifiers are reported
- duplicated sample identifiers are reported
- orientation is automatically detected
- unsupported formats are rejected

Quality Requirements:

- missing values should be minimal
- gene identifiers should be unique whenever possible
- sample identifiers should be unique
- expression matrix should contain biologically meaningful values

Output after harmonization:

All matrices are converted to:

sample x gene

for downstream analysis.

---

### 1.2 Metadata File

Purpose:

Stores sample annotations required for biological interpretation and grouping of samples.

Supported formats:

- CSV
- TSV
- XLSX

Required columns:

| Column | Description |
|----------|----------|
| sample_id | Unique sample identifier |
| group | Biological group, phenotype or condition |

Optional columns:

| Column | Description |
|----------|----------|
| dataset | Dataset source |
| subtype | Disease subtype |
| stage | Disease stage |
| batch | Experimental batch |
| tissue | Tissue type |

Example:

sample_id,group
S1,TUMOR
S2,TUMOR
S3,NORMAL

Example (extended):

sample_id,group,dataset,subtype
TCGA_001,BRCA,PANCAN,LuminalA
TCGA_002,BRCA,PANCAN,LuminalB
TCGA_003,LUAD,PANCAN,Adenocarcinoma

Validation Rules:

- sample_id must be unique
- group cannot be empty
- metadata samples should match expression matrix samples
- duplicated sample identifiers are reported
- missing metadata entries are reported

Quality Requirements:

- every sample should have metadata
- every sample should belong to a defined group
- metadata should be internally consistent

Output after harmonization:

Metadata are standardized into clean_metadata.csv.

## 5. Data Design Principle

All files must support transparency, reproducibility and traceability.

Each cleaning operation must be documented with:

- what was detected
- what was changed
- why it was changed
- which rule was applied

---

## 6. Internal Working Format

After harmonization, all expression matrices will be stored in the same internal format:

sample x gene

This means that:

- rows represent samples
- columns represent genes
- the first column is always sample_id
- all gene expression values must be numeric

Example:

sample_id,TP53,KRAS,EGFR
S1,5.2,7.1,2.3
S2,6.1,8.4,1.9

Reason:

This format is directly compatible with Python-based exploratory analysis, including PCA, clustering, heatmaps and machine learning libraries such as scikit-learn.

## 7. clean_expression_matrix.csv

Purpose:

Stores the harmonized and cleaned expression matrix generated by the Data Cleaner.

The file serves as the primary input for the Transcriptomic Analysis Engine.

Format:

sample x gene

Required characteristics:

* first column = sample_id
* rows represent samples
* columns represent genes
* all expression values must be numeric
* duplicated genes resolved according to project rules
* constant genes removed from analytical dataset
* harmonized structure across all supported datasets

Example:

sample_id,TP53,KRAS,EGFR
S1,5.2,7.1,2.3
S2,6.1,8.4,1.9
S3,5.8,7.9,2.1

Rules:

* internal project format is always sample x gene
* file must be compatible with PCA, clustering and heatmap generation
* expression values must be numeric
* output is generated only after successful harmonization and quality control

sample_id,group,dataset

TCGA_002,BRCA,PANCAN
TCGA_003,LUAD,PANCAN

- column names are standardized by Data Cleaner

## 9. audit_log.csv

Purpose:

Stores all automatic decisions and actions performed by the Data Cleaner.

The audit log provides full transparency and traceability of data cleaning operations.

Required columns:

| Column | Description |
|----------|----------|
| timestamp | Date and time of action |
| module | Module performing the action |
| action | Detected issue or performed operation |
| target | Gene, sample or column affected |
| decision | Action taken by the system |
| status | PASS, WARNING, FAIL or REQUIRES REVIEW |
| reason | Explanation of the decision |

Example:

timestamp,module,action,target,decision,status,reason

2026-06-01 10:00,Harmonization,Column Standardization,Sample ID,renamed,PASS,Column names were standardized.

2026-06-01 10:05,Missing Data,Median Imputation,TP53,imputed,WARNING,Missing values below threshold.

2026-06-01 10:10,Constant Gene Detection,Remove Constant Gene,GAPDH,removed,WARNING,Constant gene not informative for exploratory analysis.

Rules:

- every automatic action must be logged
- every entry must contain a reason
- status must use predefined project categories
- audit log must be human-readable

---

## 10. harmonization_report.csv

Purpose:

Documents all structural transformations performed during data harmonization.

Required columns:

| Column | Description |
|----------|----------|
| original_element | Original value or structure |
| standardized_element | Standardized value or structure |
| element_type | Type of transformed element |
| transformation | Applied transformation |
| reason | Reason for transformation |

Example:

original_element,standardized_element,element_type,transformation,reason

Sample ID,sample_id,column_name,renamed,Standardized technical column naming.

Condition,group,metadata_column,renamed,Unified metadata group column.

gene x sample,sample x gene,data_orientation,transposed,Analysis engine requires a standardized internal format.

Rules:

- all structural transformations must be reported
- no harmonization step can occur without documentation
- the report must be human-readable

---

## 11. data_quality_report.csv

Purpose:

Provides a structured summary of all quality control checks performed by the Data Cleaner.

Required columns:

| Column | Description |
|----------|----------|
| check | Name of quality control check |
| status | PASS, WARNING, FAIL or REQUIRES REVIEW |
| metric | Measured value |
| threshold | Applied threshold |
| details | Human-readable interpretation |

Example:

check,status,metric,threshold,details

Missing Data,WARNING,8.3%,<5%,Missing values exceed recommended threshold.

Metadata Consistency,PASS,100%,100%,All samples have matching metadata.

Duplicate Genes,WARNING,3 duplicates,0 duplicates,Duplicated genes detected and aggregated.

Constant Genes,WARNING,25 genes,0 genes,Constant genes removed from analysis dataset.

Numeric Data Check,PASS,0 invalid values,0 invalid values,All expression values are numeric.

Rules:

- every QC module must generate one summary entry
- thresholds must be reported whenever applicable
- all statuses must use predefined project categories
- report must be human-readable

---

## 12. data_readiness_report.csv

Purpose:

Provides the final assessment of dataset readiness for downstream analysis.

Required columns:

| Column | Description |
|----------|----------|
| readiness_status | Final dataset status |
| total_checks | Total number of quality control checks |
| pass_count | Number of PASS checks |
| warning_count | Number of WARNING checks |
| fail_count | Number of FAIL checks |
| review_count | Number of REQUIRES REVIEW checks |
| recommendation | Final recommendation |

Possible statuses:

- READY_FOR_ANALYSIS
- READY_WITH_WARNINGS
- REQUIRES_REVIEW

Example:

readiness_status,total_checks,pass_count,warning_count,fail_count,review_count,recommendation

READY_WITH_WARNINGS,5,3,2,0,0,Dataset can be analyzed but warnings should be reviewed.

Rules:

- exactly one final status must be assigned
- recommendation must always be provided
- readiness assessment is based on data_quality_report.csv
